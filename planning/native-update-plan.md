# NEBULOUS update and native target plan

## Local searchable workspace

`.native-code/current` is ignored and generated by:

```powershell
.\scripts\NativeCode\Export-NebulousManagedCode.ps1 -Force -PackWithRepomix
```

The script decompiles configured first-party managed assemblies, writes type and
resource indexes, records hashes/versions/timestamps, and produces a compressed
Repomix pack. `-ValidateOnly` checks paths and tools without changing output.

## Update workflow

1. Preserve the previous `manifest.json` and feature evidence summary.
2. Record game build/version visible in the install and logs.
3. regenerate the dump into a clean hash-named assembly tree;
4. compare assembly hashes, type indexes, relevant signatures, and prefab YAML;
5. run the native target verifier;
6. review all changed targets by owning feature;
7. rebuild and run API/serialization/package checks;
8. run targeted component fixtures, then broad smoke tests;
9. update the supported evidence ledger and release notes.

## Target verifier design

Input:

- a machine-independent target manifest committed under planning/compatibility;
- configured native assembly paths;
- feature owner and failure policy for each target.

Validate:

- declaring type and assembly;
- field/property/method existence;
- static/instance form and visibility;
- exact field/property type;
- method return and parameter types, ref/out/generic form, and overload;
- Harmony prefix/postfix parameter binding;
- typed `FieldRef` compatibility.

Output:

- pass/fail grouped by feature;
- old and observed signature;
- callers and accessor source;
- whether the failure blocks build, disables a feature, or requires runtime
  evidence.

## Prefab diff scope

Only dump and compare prefabs relevant to changed features. Capture component
presence/order, serialized references, runtime descriptor bindings, muzzle/effect
graphs, socket/filter data, and network identity/pool configuration.

## Current migration priorities

Use `planning/generated/reflection-migration-inventory.md`. Start with repeated
or gameplay-mutating private access in sensor, resource, socket, missile, and
weapon hot paths. Cold YAML/debug tooling can remain later. Keep dynamic
reflection only where member/runtime type is genuinely unknown.
